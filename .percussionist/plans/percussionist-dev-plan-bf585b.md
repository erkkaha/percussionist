# Plan: Fix stale parent-branch baseline for sequential BUILD task worktrees

**Task:** `percussionist-dev-plan-bf585b`  
**Primary file:** `packages/operator/src/pod-builder.ts`

---

## Context

The stale-baseline behavior is rooted in the workspace-init shell script generated by `buildPodSpec()` in `packages/operator/src/pod-builder.ts`.

Relevant existing flow:

1. Mirror update phase fetches remote branches into `refs/remotes/origin/*` and then tries to sync `refs/heads/*` from those remote-tracking refs.
2. That `refs/heads` sync intentionally skips branches currently checked out by any active worktree (`git worktree list --porcelain | grep "branch refs/heads/<branch>"`).
3. During worktree creation for feature-branch tasks, when target branch does not yet exist, the script creates it from `git.parentRef` using:
   - reused-worktree create path: `git -C "$MIRROR_DIR" worktree add -b "${git.ref}" "$WORKTREE_DIR" "${git.parentRef}"`
   - freshWorktree create path: same command pattern
4. Because `git.parentRef` resolves via local branch ref (`refs/heads/<parent>`), it can be stale if that parent branch was excluded from sync due to another active/stale worktree (e.g., buildgen or prior run worktree).

This exactly matches the reported failure mode: BUILD B is created from stale `refs/heads/feature/{plan}` even though `refs/remotes/origin/feature/{plan}` has newer commits from merged BUILD A.

## Scope boundaries

### In scope

- Workspace-init git branch creation behavior in `packages/operator/src/pod-builder.ts` for runs that set both `git.ref` and `git.parentRef`.
- Ensuring new BUILD branches derive from latest remote parent tip when available.
- Both worktree modes:
  - `worktreeReuse = true` create path (`else` branch when worktree absent)
  - `worktreeReuse = false` freshWorktree path

### Out of scope

- Branch naming/resolution logic in manager-controller (`resolveTaskBranch`, `resolveParentBranch`, merge sequencing logic).
- Task dependency gating (`predecessorRef`) and merge-run logic.
- Broader mirror synchronization redesign beyond parent-branch baseline correctness at branch-creation time.

## Approach

Use **remote-tracking parent ref as authoritative branch baseline** when creating a new branch from `parentRef`.

### Key decision

Prefer creating from `refs/remotes/origin/${git.parentRef}` (or an equivalent guarded resolution to that ref) instead of plain `${git.parentRef}`.

Why:

- `refs/remotes/origin/*` is refreshed by fetch regardless of worktree locks.
- `refs/heads/*` can remain stale by design when a branch is checked out in another worktree.
- This is the narrowest fix that preserves existing behavior and only changes fallback branch-creation baseline.

### Guard behavior for first-BUILD / missing remote branch

Keep compatibility for cases where `origin/<parent>` does not exist yet (e.g., parent branch not pushed yet, or race on first run):

1. If `refs/remotes/origin/${git.parentRef}` exists, create from that.
2. Else fall back to existing behavior (`${git.parentRef}` local ref).
3. If neither exists, keep current hard failure semantics (workspace-init exits as today).

### Suggested script shape

In each `worktree add -b ... parentRef` fallback block, add a small parent-base resolution block before creating the branch:

- Compute `_PARENT_REMOTE_REF="refs/remotes/origin/${git.parentRef}"`
- Compute `_PARENT_BASE_REF="${git.parentRef}"` as default
- If `rev-parse $_PARENT_REMOTE_REF` succeeds, set `_PARENT_BASE_REF="$_PARENT_REMOTE_REF"`
- Run `git worktree add -b ... "$_PARENT_BASE_REF"`

Also update log text to indicate whether remote-tracking or local parent baseline was used.

This avoids `update-ref` against checked-out branches and avoids force-moving local heads.

## Risks / open questions

1. **Remote-tracking ref availability timing:**
   - If fetch fails, remote-tracking refs may be stale or absent; current script already tolerates fetch failure. We should keep behavior but log clearly when fallback local ref is used.

2. **Hidden divergence between local and remote parent refs:**
   - Preferring remote can surface branch states not present in local `refs/heads`; this is desired for freshness but should be explicitly documented in comments.

3. **Dual-path drift risk (`worktreeReuse` vs `freshWorktree`):**
   - Logic exists in two near-identical blocks; both must be updated identically to avoid mode-specific regressions.

4. **Potential future simplification:**
   - Long-term, mirror ref-sync strategy could be revisited, but this plan intentionally keeps a focused fix at branch-creation time to minimize blast radius.

## Assumptions

- BUILD task branches in feature-branch mode are created by workspace-init with `git.ref` set to child branch and `git.parentRef` set to plan branch.
- `git fetch origin '+refs/heads/*:refs/remotes/origin/*'` remains the authoritative source of latest remote parent tip in mirror repos.
- No required API/schema changes are needed; this is purely operator script-generation behavior.
